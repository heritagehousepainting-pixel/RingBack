# Phase 2 — Post-Build Audit Reconciliation (Opus)
**Date:** 2026-06-18 · Two parallel Sonnet auditors (A = spec/correctness/test-integrity, B = security/honesty/risk-lane) reviewed the merged build. Suite was 35/35 green but several new tests STUBBED cross-agent functions, so green did not prove the seams. B's schema-level audit caught real bugs A's call-path trace missed. Verdict + dispositions below.

## No P0s (both auditors agree).

## P1 — all FIXED
- **A-P1 / morning-of reminder was dead code.** `enqueue_morning_reminder` was defined + tested but never called from the booking path → morning reminders never queued in production. **Fix:** wired `reminders.enqueue_morning_reminder(biz, lead, gday, gtime)` into both booking hooks (open_conversation + handle_inbound), right after `enqueue_reminder`. (app.py)
- **B-P1-A / SF-7 self-attest on prod misconfig.** When Twilio is configured but `FIRSTBACK_PUBLIC_URL` is unset, the sentinel can't be placed and the route auto-confirmed `forwarding_confirmed=True` (comment falsely said "unreachable on production") — re-introducing the exact unproven "confirmed" SF-7 removed. **Fix:** when telephony is configured but the sentinel can't be placed (or no owner cell), the route now leaves forwarding UNCONFIRMED and redirects `?unverified=1` (or `unverified=nocell`). Self-confirm only when there's no telephony at all (pure local dev). (app.py, both dial + catcher branches)
- **B-P1-B / SF-4 webhook retry path broken.** `twilio_sms_status` read `row["business_id"]` + `retry_count` from a `messages` row that has neither → KeyError (swallowed) so the primary async retry never fired; and `attempt` always computed to 1 → cap bypassed → would have looped every 30s. The retry body also embedded a `[retry_to:+1…]` prefix that `run_due_once` never stripped → the customer would receive `[retry_to:…] <message>`. **Fixes:** (1) `db.get_message_by_provider_sid` now LEFT JOINs leads to return `business_id` + `lead_phone`; (2) new `db.count_sms_retries(lead_id, within_minutes=30)` derives the attempt + enforces the 3-cap from the webhook (the messages table can't track retry depth); (3) `db.queue_sms_retry` stores the CLEAN body (destination resolved from the lead at send time). Added real (un-stubbed) coverage in test_sf4_db.py for the enrichment + cap + clean body — the exact gap the stubbed tests left. The `run_due_once` direct-failure path was already correct.

## Dispositioned (not changed)
- **B-P1-C / due_scheduled_messages INNER JOIN drops NULL-lead retry rows.** Inert in practice: retry rows are only ever created with a lead_id (the webhook now guards `row.get("lead_id")`; run_due_once rows already passed the same INNER JOIN, so they have a lead). A NULL-lead retry row is never created. Changing the shared Phase-1 query (used broadly by reminders) to a LEFT JOIN is riskier than the latent issue. **Left as-is, documented.**
- **A-P2 / cross-agent stubs in F04 + SF7 tests.** Acceptable per the agreed build model; added real db-layer coverage for the load-bearing functions.
- **B-P2-A / "simulated"/unverified state not surfaced in UI.** The route now emits `?unverified=1` / `nocell`; the /setup banner copy for it is deferred polish (the state is honest — forwarding stays "pending", never "confirmed").
- **B-P2-B / sms_fail alert body renders thin (ids, not name).** Owner-facing cosmetic; the alert still conveys "a text failed after retries." Deferred.
- **B-P2-C / morning-of near-term ordering.** Not a bug — `enqueue_morning_reminder` skips when the estimate hour <10 and when the morning has already passed, and dedupes via find_scheduled_message.

## Result: all 7 Phase 2 items implemented + wired (no dead code), honesty contract holds across both forwarding modes AND the misconfig path, SF-4 retry works end-to-end and is bounded. **35/35 test files green** after fixes.
